# 开发指南 (Developer Guide)

## 1. 技术栈与环境

- **语言标准**: C++20
  - **GCC**: 10.0+ (推荐 11.2+)
  - **Clang**: 12.0+ (推荐 14.0+)
  - **MSVC**: VS2019 16.11+
- **构建系统**: CMake 3.20+
- **包管理**: vcpkg (推荐) 或 Conan
- **核心依赖**:
  - `boost`: Asio (网络), Beast (WebSocket), CircularBuffer
  - `fmt`: 格式化日志
  - `nlohmann/json`: JSON 序列化
  - `spdlog`: 高性能日志
  - `openssl`: 加密签名
  - `prometheus-cpp`: 指标暴露

## 2. 环境搭建 (MacOS/Linux)

```bash
# 1. 安装 vcpkg
git clone https://github.com/microsoft/vcpkg
./vcpkg/bootstrap-vcpkg.sh

# 2. 安装依赖
./vcpkg/vcpkg install boost fmt nlohmann-json spdlog openssl prometheus-cpp

# 3. 克隆项目
git clone <your-repo-url>/ai-trade.git
cd ai-trade
```

## 3. 构建与编译

```bash
# 配置 CMake (假设 vcpkg 在上级目录，使用 -S 源目录 -B 构建目录 的现代语法)
cmake -S . -B build -DCMAKE_TOOLCHAIN_FILE=../vcpkg/scripts/buildsystems/vcpkg.cmake

# 编译
cmake --build build -j 8
```

### 4.1 回放模式 (Replay)

```bash

  --exchange=bybit \
  --replay_market_data=data/reports/replay_validation/latest/segment_01/replay_market.csv \
  --replay_timestamp_column=timestamp \
  --replay_symbol_column=symbol \
  --replay_price_column=price \
  --replay_volume_column=volume \
  --replay_interval_column=interval_ms \
  --replay_funding_rate_column=funding_rate_per_interval \
  --replay_default_interval_ms=300000
```

如果希望直接从归档特征数据中抽取 `TREND` 片段并完成 replay 验证，可运行：
```bash
python3 tools/run_replay_validation.py \
  --feature_csv=data/research/feature_store_5m.tmp.csv \
  --feature_csv_by_symbol=SOLUSDT=data/research/SOLUSDT/feature_store_5m.csv,XRPUSDT=data/research/XRPUSDT/feature_store_5m.csv \
  --base_config=config/bybit.replay.assess.maker_first.yaml \
  --corpus_manifest=data/research/replay_validation_trend_corpus.json \
  --source_symbol=BTCUSDT \
  --symbols=SOLUSDT,XRPUSDT \
  --output_dir=data/reports/replay_validation/latest
```
输出目录会生成：
- `replay_validation_report.json`
- `replay_optimization_report.json`
- 单币对模式：`segment_*/replay_market.csv` / `runtime.log` / `runtime_assess.json`
- 多币对模式：`<SYMBOL>/segment_*/replay_market.csv` / `runtime.log` / `runtime_assess.json`

`replay_validation_report.json` 现在还会补一层聚合结果：
- `aggregate_summary`
- `aggregate_validation`
- `selection`
- `symbols`
- `real_market_replay`
- `per_symbol_source`
- `symbol_reports`
- `execution_economics`
- `cost_sensitivity`
- `exit_capture`
- `execution_cost_plan`
- `execution_optimizer`

`execution_economics` 会把每个 replay segment 的 fills、扣费后
`realized_net_per_fill`、估算手续费、主单/probe fill 数、maker/taker fill 数、
趋势片段强度和流动性等指标合并到同一层，便于判断“亏损来自没有成交、成交后全亏、
手续费吞噬，还是只在弱趋势片段亏损”。
每个 segment 还会补充 close-path 方向、`close_path_mfe/mae` 与
`close_path_efficiency`，用于粗定位“行情给过空间但退出没吃到”还是“片段本身没有足够路径收益”。

`exit_capture` 会把有成交片段的路径 MFE、手续费 bps、估算 gross 与 path-potential
合并成退出捕获诊断；重点看 `primary_diagnosis`：
`exit_capture_low` 表示行情给过足够空间但 gross 捕获不足，应优先改退出/盈利保护；
`fee_bound_or_low_path` 表示路径空间本身难以覆盖费用，应优先减少交易或提高入场净 edge。

`execution_cost_plan` 会基于当前 replay fills 给出是否值得重跑低成本执行方案：
`status=candidate_requires_rerun` 只代表 maker-first/低费路径在诊断上可能转正，
仍必须用新执行配置重跑 replay/full，不能直接激活模型。

`cost_sensitivity` 是诊断项：它基于同一批 replay 结果重算不同费用倍率
（如 `fee_x0.5 / fee_x0.75 / fee_x1`）和 gross-edge 过滤后的净值。
`status=diagnostic_pass` 只表示在某些假设成本下可能转正，不能作为上线通过；
上线仍以当前成本下的 `aggregate_validation` 和 `execution_optimizer` 为准。
重点看 `current_cost_status`、`break_even_fee_multiplier` 和 `fee_x1` 场景。

`execution_optimizer` 是轻量离线诊断器：它不会重新跑交易程序，而是基于本轮已完成的
replay segments 搜索若干预过滤候选，例如更高趋势强度、更高流动性、更低区间波动等。
若 `status=fail` 且原因包含
`no_deployable_prefilter_candidate_positive_after_costs`，表示本轮没有任何基于片段特征的
可部署过滤候选能在扣除成本后转正，下一步应优先改训练目标、退出策略或成本模型，而不是继续放宽入场门槛。
`diagnostic_*` 候选只用于解释问题，因为它们会使用事后运行指标，不应直接上线。

当 live 报告显示 `sections.replay_symbol_alignment` 未覆盖 live `TREND/TREND_CANDIDATE`
币对时，应优先使用 `recommended_replay_symbols_csv` 作为 `--symbols` 或
`CLOSED_LOOP_REPLAY_VALIDATION_SYMBOLS`，让下一轮 replay 覆盖本轮真实 TREND/CANDIDATE；
聚合报告会同时输出全局覆盖结果和每个 symbol 的 `aggregate_validation`。
若没有提供 `--feature_csv_by_symbol`，且 `--source_symbol` 与 `--symbols` 不一致，报告会保留告警：
这只能证明执行链路和 Universe 配置能覆盖目标币对，不能替代该币对自己的历史行情 feature store 验证。
当 `--feature_csv_by_symbol` 覆盖所有目标币对时，报告会输出 `real_market_replay=true`。

其中 `selection.selection_mode` 用来区分本次 replay 是：
- `corpus_manifest`：复用了固定趋势语料；
- `corpus_manifest_plus_dynamic`：先复用固定趋势语料，再用当前 feature store 的新片段补足覆盖；
- `dynamic_top_n`：当前没有可用语料，按最新 feature store 动态选段并回写 manifest。

`config/bybit.replay.assess.maker_first.yaml` 是当前闭环默认 replay profile：
它沿用“接近 live/S5 的 execution 与 strategy 主参数”，但把入场按 maker-first
低成本路径估算，出场仍按 taker 成本保守计费，用于验证最近
`execution_cost_plan.status=candidate_requires_rerun` 指向的低成本执行假设。
`config/bybit.replay.assess.yaml` 保留为 current-cost/taker-like 基线，适合做对照回放。
两者都保留 replay 特有的短窗口适配（如更低的 `adaptive_fee_gate_min_samples`），
避免因为片段太短而完全失真。
Replay 适配器会把 `SL/TP` 保护单作为条件单挂起，只有后续行情穿越
trigger price 才生成 reduce-only 成交，因此默认 maker-first profile 已可以验证
“低成本执行 + 保护单退出”组合，而不是把保护单错误地当成立即成交单。
Bybit replay 的本地虚拟成交会按 `execution.entry_fee_bps / exit_fee_bps` 计入
`FillEvent.fee`，并对最终会按 Market/Taker 执行的成交按
`execution.expected_slippage_bps` 做不利方向滑点调整；因此 replay 的
`realized_net_per_fill` 不再只是“是否成交”的零值占位，而是包含手续费和 taker
滑点后的执行经济性样本。

脚本现在会按 `available_segments` 的优先级自动回放，优先级综合考虑片段长度、趋势强度和成交量，并优先尝试达到更稳健的覆盖建议：
- 最小门槛：`min_execution_active_runs`
- 最小门槛：`min_execution_pass_runs`
- 最小门槛：`min_total_fills`
- 建议门槛：至少 `4` 个真正 `EXECUTION_ACTIVE` 片段（即出现 submit/enqueue/fill，而非仅策略非零参与）
- 建议门槛：至少 `4` 个 `execution_status=PASS` 片段
- 建议门槛：至少 `6` 个总 fills

达到建议门槛后会提前停止；若只达到最小门槛，脚本会在 `max_segments` 范围内继续扩样，避免“刚够 3 fills 就停”导致证据过薄。可通过 `selection.minimum_coverage_targets_met`、`selection.recommended_coverage_targets_met`、`aggregate_validation.coverage_strength_status` 和 `selection.segments_ran` 判断这次 replay 的证据强度。

其中 `aggregate_validation.status` 的语义为：
- `pass`：趋势回放片段里已经拿到足够 execution 样本，且聚合 `realized_net_per_fill` 过线
- `pass_with_actions`：execution 已启动，但 `ORDER_FILTERED_COST`、单段 `PASS_WITH_ACTIONS` 偏多，或已有 fills 但 `realized_net_per_fill` 全为 `0`，适合继续做 entry gate / cost filter / 扣费净值口径定向优化
- `fail`：趋势回放样本不足，或聚合 execution 经济性仍未过线；默认要求 `mean_realized_net_per_fill >= 0.0`

多币对 replay 额外输出 `aggregate_validation.symbol_tradeability`：
- `tradable_symbols`：覆盖达标且均值净收益过线的主链可交易白名单
- `quarantined_symbols`：覆盖已足够但扣费后经济性为负的隔离观察币对
- `insufficient_symbols`：覆盖不足，不能下经济性结论的币对

聚合层允许“白名单通过、失败币种隔离”：只要 `tradable_symbol_count >= --min_tradable_symbols`，
经济性失败但覆盖充分的币对会进入 `quarantined_symbols`，不会再把整个
`replay_validation.status` 拉成 `fail`。覆盖不足仍然是 hard fail，因为它代表证据不够，而不是币种质量差。

注意：`coverage_strength_status` 只表示覆盖强度，不再代表经济性通过。
因此报告可能出现 `coverage_strength_status=ROBUST` 但 `status=fail`，这表示趋势样本足够，
但扣除手续费/滑点后的净质量仍不达标，应优先修 execution economics，而不是继续扩样。
当所有有成交片段都未出现正 `realized_net_per_fill` 时，即使兼容旧阈值也会被判定为质量失败。

如果 `aggregate_summary.zero_realized_net_with_fills_runs > 0`，说明 replay 已经能触发成交，
但部分或全部成交片段没有形成扣费后的净值样本。特别是
`nonzero_realized_net_with_fills_runs=0` 时，只能证明“执行和保护链路可触发”，不能证明
策略在扣除手续费、滑点和平仓成本后具备经济性。

其中 `aggregate_validation.coverage_strength_status` 的语义为：
- `ROBUST`：不仅达到最小门槛，也达到了建议覆盖门槛
- `MINIMUM_ONLY`：已达到最小门槛，但趋势 execution 样本仍偏薄
- `INSUFFICIENT`：连最小门槛都未达到

如需调整 replay 聚合门槛，可额外传入：
- `--min_execution_active_runs`
- `--min_execution_pass_runs`
- `--min_total_fills`
- `--min_mean_realized_net_per_fill`
- `--warn_mean_filtered_cost_ratio`
- `--min_tradable_symbols`

如需强制重建语料，可额外传入：
- `--refresh_corpus_manifest`

`tools/closed_loop_runner.sh full|train|data` 现在也可以自动接入这条链路：
- 当 `feature_store_5m.csv` 可用时，会自动生成 `replay_validation_report.json`
- 并在 `closed_loop_report.json` 中输出 `sections.replay_validation`
- 顶层同时补充 `replay_readiness_status`

如果当前 run 没有 feature store（例如数据链路回退到仅 `R0 fetch`），runner 会把 replay 标记为 `pass_with_actions` 的 skipped，而不会直接打崩整条闭环。

```bash
  --label_round_trip_cost_bps=13.0 \
  --label_min_net_edge_bps=1.3 \
  --feature_clip_quantile=0.001
```

Docker 方式（推荐，避免本机依赖不一致）：

```bash
docker compose run --rm ai-trade-research \
  --csv=./data/research/ohlcv_5m.csv \
  --miner_report=./data/research/miner_report.json \
  --output=./data/research/integrator_report.json \
  --model_out=./data/models/integrator_latest.cbm
```

验收要点：
- 日志出现 `INTEGRATOR_START` / `INTEGRATOR_SPLIT` / `INTEGRATOR_DONE`
- 报告包含 `model_version`、`feature_schema_version`、`metrics_oos`、`feature_importance`
- `metrics_oos.splits` 中每个 split 都有明确的 train/test 时间窗口
- 产物中包含基线对照：`baseline_auc` 与 `delta_auc_vs_baseline`
- 训练口径固定为 `t` 特征预测 `t+1...t+h`（反前视）
- 报告 `data.label_policy` 会记录成本带标签口径：成本带内样本会被丢弃，避免模型学习“方向正确但扣费后无净收益”的交易。
- 报告 `feature_transform.clip_bounds` 会记录训练分布裁剪边界、中心值和缩放尺度，线上 Integrator 会按同一参数裁剪并归一化特征，避免 `miner_*` 大尺度异常值直接进入 CatBoost。

### 4.2.3 闭环编排（R0/R1/R2 + 运行验收 + 汇总报告）

项目已提供一键闭环脚本：`tools/closed_loop_runner.sh`。

本地执行示例：
```bash
# 训练闭环：R0 -> R1 -> R2 -> replay validation -> 模型注册 -> 汇总报告
tools/closed_loop_runner.sh train

# 部署后短窗口冒烟：导出日志 -> SMOKE 判定 -> 汇总报告
tools/closed_loop_runner.sh assess --stage SMOKE --since 15m

# 运行态验收闭环：导出日志 -> S5 判定 -> 汇总报告
tools/closed_loop_runner.sh assess --stage S5 --since 4h

# 全流程闭环
tools/closed_loop_runner.sh full --stage S5 --since 4h

# 数据加速链路
tools/closed_loop_runner.sh data --data-config config/data_pipeline.yaml

# 产物回收（可手动执行）
tools/recycle_artifacts.sh \
  --reports-root ./data/reports/closed_loop \
  --keep-run-dirs 120 \
  --keep-daily-files 120 \
  --keep-weekly-files 104
```

说明：
- `train/full` 会先执行 `D1 baseline freeze`（冻结当前 active 模型快照）
- 紧接执行 `D2 data quality gate`（样本不足/脏数据会直接阻断训练）
- `data` 会执行归档下载、增量同步、缺口回补、特征构建与 walk-forward 回测
- 产物分别写入：`baseline_report.json`、`data_quality_report.json`
- 每次运行结束会自动执行产物回收，防止报告目录无限增长

## 7. 常见问题排查（Bybit）

- 报错：`retCode=10001, Qty invalid`
  - 当前实现会在连接后读取 `GET /v5/market/instruments-info`，并按交易规则自动处理下单数量：
    - 按 `qtyStep` 向下量化
    - 校验 `minOrderQty`
    - 校验（非 reduce_only）`minNotionalValue`
  - 系统会将 `candidate_symbols + fallback_symbols + primary_symbol` 与交易所 `symbol` 规则做交集过滤：
    - 非 `TRADING` 状态会被过滤
    - 若获取到规则但全部被过滤，进程会拒绝启动（避免“全是不可交易币对”）
  - 若仍出现该错误，优先检查：
    - 配置中 `execution.max_order_notional` 是否过小（导致量化后低于最小下单量）
    - 目标 symbol 是否在当前账户可交易
    - 日志中是否有“数量低于最小下单数量/最小名义金额”的拒单提示
