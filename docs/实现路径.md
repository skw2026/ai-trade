# AI 自闭环实现路径（先跑通，再优化）

## 1. 目标与优先级

当前执行顺序固定为两步：
1. 先确保自闭环流程稳定可运行（可持续执行、可验收、可审计）。
2. 再基于运行报告做工程优化（按数据驱动，不凭感觉改参数）。

本文件对应当前仓库实现（`tools/closed_loop_runner.sh` + `docker-compose.prod.yml` + `ops/watchdog.py`）。
新增数据加速链路入口：`tools/closed_loop_runner.sh data`（详见 `docs/数据加速链路.md`）。

### 1.1 当前项目进度（2026-04-11）

当前项目已经不在“架构是否成立”的阶段，而是在“执行质量样本是否充分、不同市场桶是否被正确解释”的阶段。

已完成：
- 闭环主链稳定：`closed_loop`、报告汇总、CD 部署门禁、回滚、watchdog/scheduler 均已打通。
- 离线研究链路稳定：`walkforward`、`registry`、`integrator`、`miner`、`data_quality` 已能持续产出可用结果。
- `registry` 泛化治理已稳定通过：当前已不再卡在 `train_test_auc_gap_mean` 或 `random_label_auc` 单次偶发超线。
- 运行态保护逻辑稳定：`policy-flat` 已从“误判低活跃故障”改为显式的保护通过语义。
- `silent flat` 语义已补齐：`STR_FLAT_SIGNAL + REG_RANGE/REG_EXTREME` 会按坏桶主动空仓解释，
  但“分支互相抵消导致净额为 0”的窗口不会被误算成 `policy-flat`。
- 反退化门禁已上线：防止系统通过长期不交易伪装成“已收敛”。
- `trend_validation` 已加入闭环汇总：可基于离线 `TREND` 桶表现独立回答“趋势样本一旦出现，当前策略是否值得进入 execution 验证”。
- `TREND replay` 验证链已具备：可从归档特征数据中抽取趋势片段，生成历史行情 replay CSV，并驱动 `trade_bot` 做本地 execution 验证。
- `TREND replay` 现已补充聚合判定：可直接回答“趋势 execution 是否已经达到继续调参/继续上线验证的最小门槛”，避免逐段手工解读。
- `TREND replay` 会自动扩展片段覆盖，直到满足最小 execution/fill 门槛或跑到片段上限，减少手工调 `max_segments` 的无效工作。
- `TREND replay` 现已支持固定语料 manifest：闭环优先复用同一批趋势片段做回放，只有 manifest 缺失或显式刷新时才会重新动态选段，避免每天因样本池波动而误判。
- `closed_loop full/train/data` 现已可选接入 `replay_validation` section：当特征数据可用时，会自动运行 `tools/run_replay_validation.py`，并将趋势 replay 聚合结果写入 `closed_loop_report.json`。
- 当前闭环默认会优先扩大 replay 覆盖：`max_segments=16`、`min_segment_bars=40`，先尽量补足趋势 execution 样本，再决定是否需要继续放宽门槛。

当前主状态：
- `TREND` 桶在离线评估中仍是正向桶；
- `RANGE/EXTREME` 桶仍是负向桶，因此线上在这类窗口主动空仓是合理行为；
- 若运行窗口没有 `TREND` 样本，runtime 结果应解释为“保护逻辑通过，但执行质量暂未验证”，而不是“策略已经完全通过”。
- 当 `trend_readiness_status=PASS` 且 `runtime.execution_status=NOT_EVALUATED` 时，应理解为“离线已准备好，当前只是在等 live 趋势样本”。

当前主要待解问题：
- 获取新的 `TREND` 运行样本，用于验证 execution 主链是否真正通过；
- 当 live 长时间缺少趋势样本时，优先使用 `tools/run_replay_validation.py` 做离线趋势 execution 回放，避免继续做无效调参；
- 读取 replay 报告时，优先看 `aggregate_validation.status`，再看 `aggregate_summary.mean_realized_net_per_fill` 与 `mean_filtered_cost_ratio_avg`，不要只盯单个片段；
- 隔离 demo 账户外部权益噪声，避免 `equity_change_usd` 污染运行态结论；
- 在 `RANGE` 主导市场中避免继续做无意义参数微调，优先做报告语义和样本覆盖对齐。
- 对 `PASS_WITH_ACTIONS` 结果，优先区分“报告语义噪声”与“真实执行退化”，不要把
  `NOISY_WHILE_FLAT` 或 silent-flat 口径问题继续误当成策略 alpha 问题。

报告阅读约定（当前版本）：
- `overall_status` 仍表示“整条闭环是否全部通过”，因此会同时受 `runtime / walkforward / registry` 影响。
- `runtime_health_status` 只表示运行态健康与 runtime 验收结果。
- `promotion_readiness_status` 只表示模型是否具备晋级/激活条件。
- `trend_readiness_status` 只表示离线 `TREND` 桶是否已满足最小交易/收益门槛，可视为“等待 live 趋势样本前的准备度”。
- `replay_readiness_status` 只表示 `TREND replay` 聚合 execution 是否已达到最小验证门槛，可视为“在 live 趋势样本缺席时的 execution 替代验证结果”。
- 当 `integrator.mode=shadow` 时，最常见的组合是：
  - `runtime_health_status=PASS/PASS_WITH_ACTIONS`
  - `promotion_readiness_status=FAIL`
  这不代表线上主链不可运行，而是“线上仍稳定，但新模型暂不具备晋级条件”。

## 2. P0：先把自闭环跑起来

### 2.1 最小前置条件

- 目录：代码部署目录存在（建议 `/opt/ai-trade`）。
- 环境文件：`/opt/ai-trade/.env.runtime` 存在，至少包含：
  - `AI_TRADE_BYBIT_DEMO_API_KEY`
  - `AI_TRADE_BYBIT_DEMO_API_SECRET`
- 可选但建议设置：
  - `AI_TRADE_IMAGE`（未设置时默认 `ai-trade:latest`）
  - `AI_TRADE_RESEARCH_IMAGE`（未设置时默认 `ai-trade-research:latest`）
  - `AI_TRADE_PROJECT_DIR=/opt/ai-trade`（用于 scheduler 二次 compose 路径对齐）
  - `AI_TRADE_ENV_FILE=.env.runtime`（scheduler 读取的 env 文件名）

### 2.2 启动顺序（生产）

```bash
cd /opt/ai-trade
set -a && source .env.runtime && set +a

# 建议固定绝对路径，避免 Docker socket 二次 compose 时路径漂移
export AI_TRADE_PROJECT_DIR=/opt/ai-trade
export AI_TRADE_ENV_FILE=.env.runtime

# 启动交易主服务 + 看门狗 + 调度器
# scheduler 会周期执行 full（train + assess）
docker compose -f docker-compose.prod.yml --env-file .env.runtime up -d ai-trade watchdog scheduler
```

### 2.3 首次验收（必须执行）

```bash
# 一键体检（容器 + scheduler/watchdog 日志 + latest 报告）
tools/check_runtime_health.sh \
  --compose-file docker-compose.prod.yml \
  --env-file .env.runtime \
  --log-since 6h

# 1) 运行态健康
docker ps --format 'table {{.Names}}\t{{.Status}}' | rg 'ai-trade|ai-trade-watchdog|ai-trade-scheduler'

# 2) 看门狗日志（应看到 All systems operational）
docker logs --tail 200 ai-trade-watchdog

# 3) 调度器日志（应看到 closed loop full 执行记录）
docker logs --tail 200 ai-trade-scheduler

# 4) 报告产物（固定入口）
ls -lah data/reports/closed_loop/latest_closed_loop_report.json
ls -lah data/reports/closed_loop/latest_runtime_assess.json
ls -lah data/reports/closed_loop/latest_run_meta.json
```

### 2.4 P0 通过标准

- `ai-trade`、`ai-trade-watchdog`、`ai-trade-scheduler` 持续运行无频繁重启。
- `latest_runtime_assess.json` 有有效 `verdict`。
- `latest_closed_loop_report.json` 可解析且包含 `overall_status`。
- 近 24h 至少产生一次完整 run 目录：`data/reports/closed_loop/<UTC_RUN_ID>/`。

### 2.5 常见失败与定位

- `scheduler` 报路径错误：优先检查 `AI_TRADE_PROJECT_DIR` 是否为宿主机绝对路径。
- `watchdog` 无法检查容器：检查 `/var/run/docker.sock` 挂载与权限。
- `full` 失败但服务存活：先看 `ai-trade-scheduler` 日志，再看本次 run 目录内 `runtime.log` 与 `runtime_assess.json`。
- research 相关失败：检查 `AI_TRADE_RESEARCH_IMAGE` 是否可拉取且包含 `tools/*` 依赖。

### 2.6 `docker-compose.yml` vs `docker-compose.prod.yml`（闭环相关）

| 维度 | `docker-compose.yml`（开发） | `docker-compose.prod.yml`（生产） |
| --- | --- | --- |
| `ai-trade` 来源 | 本地 `build` | `AI_TRADE_IMAGE` 预构建镜像 |
| `ai-trade-research` 来源 | 本地 `Dockerfile` 的 `research` target 构建 | `AI_TRADE_RESEARCH_IMAGE`（由 `Dockerfile` 的 `research` target 构建，`profile: research`） |
| 自闭环常驻服务 | 无 `scheduler/watchdog` | 内置 `scheduler` + `watchdog` |
| 数据卷路径策略 | 直接 `./...` | `AI_TRADE_PROJECT_DIR` + 宿主绝对路径对齐 |
| 推荐用途 | 本地开发与调试 | 部署后持续闭环运行 |

规则：
- 本地调试 `train/assess/full` 可直接用 `docker-compose.yml`。
- 线上持续闭环必须使用 `docker-compose.prod.yml`，并固定 `AI_TRADE_PROJECT_DIR`。

### 2.7 CI/CD 强闭环（默认开启）

`main` 分支 CD 现在是“部署 + 运行态闭环门禁”双门槛，顺序如下：
1. Build/Test 通过后推送镜像。
2. ECS 部署新镜像并等待 `ai-trade` 健康检查。
   - 默认先停止 `watchdog`、`scheduler`，仅启动 `ai-trade` 做门禁。
   - 门禁通过后再拉起 `watchdog`、`scheduler`。
3. 健康后立即执行 `tools/closed_loop_runner.sh assess`（默认 `DEPLOY + 30m`；若运维变量改成 `full`，仍只用 runtime verdict 作为 DEPLOY 门禁）。
4. `DEPLOY` 阶段仅根据 `data/reports/closed_loop/latest_runtime_assess.json` 的 `verdict` 决定是否放行：
   - `CLOSED_LOOP_STRICT_PASS=true`：要求 `verdict=PASS`
   - `CLOSED_LOOP_STRICT_PASS=false`：允许 `PASS` / `PASS_WITH_ACTIONS`
5. `data/reports/closed_loop/latest_closed_loop_report.json` 的 `overall_status`、`registry`、`walkforward` 在 `DEPLOY` 阶段保留为审计输出，不再因离线段失败错误触发回滚。
6. 若 runtime verdict 不满足门禁，自动回滚到上一个镜像并使 CD 失败。
7. CD 自动回传 `latest_closed_loop_report.json` / `latest_runtime_assess.json` / `latest_run_meta.json` / `latest/runtime.log` 为 artifact，便于审计留痕。

安全建议：
- 建议配置 `ECS_HOST_FINGERPRINT`（Secrets 或 Variables），用于在 CI/CD 中校验 SSH 主机身份，避免中间人风险。

可通过仓库 `Actions Variables` 调整门禁参数：
- `CLOSED_LOOP_ENFORCE`（默认 `true`）
- `CLOSED_LOOP_ACTION`（默认 `assess`）
- `CLOSED_LOOP_STAGE`（默认 `DEPLOY`）
- `CLOSED_LOOP_SINCE`（默认 `30m`）
- `CLOSED_LOOP_MIN_RUNTIME_STATUS`（默认空，使用阶段内置阈值）
- `CLOSED_LOOP_OUTPUT_ROOT`（默认 `/opt/ai-trade/data/reports/closed_loop`）
- `CLOSED_LOOP_STRICT_PASS`（默认 `true`）
- `DEPLOY_SERVICES`（默认 `ai-trade watchdog scheduler`）
- `GATE_DEFER_SERVICES`（默认 `watchdog scheduler`）

可通过运行环境变量调整 S5 自学习门禁力度（`closed_loop_runner.sh`）：
- `CLOSED_LOOP_VERIFY_S5_EVOLUTION_SWITCHES`（默认 `true`；校验 `use_virtual_pnl + factor-IC + learnability` 已启用）
- `CLOSED_LOOP_REQUIRE_S5_FACTOR_IC_ACTION`（默认 `false`；强制要求 factor-IC 有实际调权动作）
- `CLOSED_LOOP_REQUIRE_S5_LEARNABILITY_ACTIVITY`（默认 `false`；强制要求 learnability 有 pass/skip 活动）
- `CLOSED_LOOP_S5_MIN_EFFECTIVE_UPDATES`（默认 `1`；强制要求 `self_evolution_effective_update_count` 达标）
- `CLOSED_LOOP_S5_MIN_REALIZED_NET_PER_FILL_USD`（默认 `-0.005`；执行质量下限）
- `CLOSED_LOOP_S5_MIN_REALIZED_NET_PER_FILL_WINDOWS`（默认 `10`；执行质量门槛生效所需最小 fills 窗口数）
- `CLOSED_LOOP_S5_MIN_EQUITY_CHANGE_USD`（默认未设置；可选强门禁：权益变化下限）
- `CLOSED_LOOP_S5_MIN_EQUITY_CHANGE_SAMPLES`（默认 `0`；权益门槛生效所需最小账户采样数）
- `CLOSED_LOOP_S5_MAX_EQUITY_VS_REALIZED_GAP_USD`（默认未设置；可选强门禁：`|equity_change - realized_net_change|` 上限）

`closed_loop_runner.sh` 的 latest 指针规则：
- `assess/full`：会刷新 `latest`、`latest_runtime_assess.json`、`latest_closed_loop_report.json`、`latest_run_meta.json` 等全局指针。
- `train`：仅产出当前 `run_dir` 报告，不刷新全局 `latest_*` 运行验收指针，防止训练产物覆盖发布门禁基线。

## 3. P1：基于运行报告做优化

### 3.1 优化输入只认报告

每轮优化前先读取：
- `data/reports/closed_loop/latest_closed_loop_report.json`
- `data/reports/closed_loop/latest_runtime_assess.json`
- `data/reports/closed_loop/latest/runtime.log`
- `data/reports/closed_loop/latest_run_meta.json`

核心观察维度：
- 运行态：`overall_status`、`runtime_verdict`、Gate fail 原因。
- 运行态分层：`runtime_validation_mode`、`protection_status`、`execution_status`。
- 资金结果：`account_outcome.equity_change_pct`、`max_drawdown_pct_observed`。
- 执行质量：成交活跃度、节流命中率、`reduce_only/gate_halted` 触发情况。
- S5 强闭环阻断：`runtime_assess` 需满足“至少 1 个 `GATE_CHECK_PASSED` + 平仓起跑口径 + 至少 1 个非零策略窗口（`strategy_mix.samples>0`）”；
  若窗口起点非平仓，会自动重定位到首个 `abs_notional <= 50` 的样本继续评估，并在指标中记录 `flat_start_rebase_applied_count=1`；
  若无可重定位样本则 FAIL。
- 若窗口以 `policy-flat` 为主，则本轮可判定为 `protection_status=PASS`，
  但 `execution_status` 必须标记为 `NOT_EVALUATED`；这类结果只能说明
  “保护逻辑通过”，不能等价为“执行质量已验证”。
- 反退化约束：若 `TREND` 桶窗口达到最小门槛，runtime 必须形成策略参与或执行样本；
  walk-forward 也必须满足 `TREND` 桶最小交易次数，避免系统在应交易行情中长期躺平。
- S5 学习有效性与执行质量强门禁：当学习动作充分但 `self_evolution_effective_update_count` 不达标，或 fills 窗口达到门槛且 `realized_net_per_fill` 低于阈值时，直接 FAIL（阈值由 `CLOSED_LOOP_S5_*` 变量控制）。
- S5 可选权益门禁：可额外启用 `CLOSED_LOOP_S5_MIN_EQUITY_CHANGE_USD` 与 `CLOSED_LOOP_S5_MAX_EQUITY_VS_REALIZED_GAP_USD`，把验收结果与账户经济结果对齐。

### 3.2 优化节奏（建议周循环）

1. 固定 3-7 天窗口收集闭环报告。
2. 仅选择一个主问题进入迭代（例如高回撤或低活跃）。
   - 若问题来自 `RANGE/EXTREME` 的策略主动空仓，不应直接把它当成系统故障；
     需要通过 `policy-flat` 口径区分“策略保护性不交易”和“系统装死”。
3. 小步改动（参数/策略/执行其一），保持可归因。
4. 再跑一轮 `assess/full`，对比关键指标再决定保留或回滚。

### 3.3 数据加速闭环（并行通道）

在 P0 稳定运行期间，可并行启用数据加速链路，缩短学习和回测迭代周期：

```bash
tools/closed_loop_runner.sh data \
  --compose-file docker-compose.prod.yml \
  --env-file .env.runtime \
  --data-config config/data_pipeline.yaml
```

该动作会依次执行：
1. Binance 归档下载
2. Bybit 增量同步
3. 缺口回补
4. 特征构建
5. walk-forward 回测

调度器也支持 `SCHEDULER_ACTION=data`，可用于每 6h/12h 自动更新训练与回测数据。

### 4.3 仓位退出体系对齐清单

当前代码与文档约定采用“三层退出并存”，后续 review / 报告排障必须按这三层核对：

1. 策略退出
   - 主链路仍是 `Strategy -> Risk -> Execution` 的目标仓位收缩。
   - 运行时表现为 `kReduce` / `reduce_only=true` 的净仓位调整。
   - 相关实现：`src/strategy/strategy_engine.cpp`、`src/system/trade_system.cpp`、`src/execution/execution_engine.cpp`

2. 动态保护单
   - `execution.protection.enabled=true` 后，系统按 symbol 总持仓维护一组 `SL/TP`。
   - 仓位变化后会基于最新均价重挂保护单；若开启 `dynamic_distance_enabled`，则用最新 `regime.volatility_level` 动态扩缩 `SL/TP` 距离。
   - 相关实现：`src/app/bot_app.cpp`、`src/execution/execution_engine.cpp`

3. 盈利保护
   - 当前已实现 `break-even` 与 `trailing SL`，仅上调 `SL`，不覆盖策略退出主链。
   - 更新条件：需要有新的有利价格极值；仅当新 `SL` 比当前 `SL` 更紧且超过最小更新阈值时才替换。
   - 相关实现：`src/app/bot_app.cpp`、`src/execution/execution_engine.cpp`

当前边界：
- 盈利保护只调整 `SL`，`TP` 仍采用初始/动态距离。
- 保护单按 symbol 总持仓维护，不做多 entry-order 粒度拆分。
- 若 `execution.protection.enabled=false`，系统只保留“策略退出”这一层。

### 4.5 Regime-Aware 调优优先级

当前闭环调优不再优先做“全局参数一把梭”，而是按 bucket 拆开：

1. `TREND`
   - 保留主策略与主仓位贡献，避免为了修 `RANGE/EXTREME` 噪声把趋势段一起收死。

2. `RANGE`
   - 优先压低弱边际意图，主要手段：
     - `strategy.range_min_confidence`
     - `strategy.trend_range_scale`
     - `strategy.defensive_range_scale`
     - `strategy.eth_range_defensive_scale_multiplier`
   - 目标不是只拦新单，而是让已有趋势仓位在 `RANGE` 里主动降档。

3. `EXTREME`
   - 优先按“不交易 / 极低参与度”处理，而不是继续尝试高成本成交。
   - 当前最直接的实现开关：
     - `strategy.trend_extreme_scale=0`
     - `strategy.extreme_block_signals=true`

### 4.4 S5 主链分类

保护体系已按 `S5 必检项 / 观测项 / 非主链项` 拆分，避免在 runtime 收敛阶段一次塞入过多变量：

1. `S5 必检项`
   - 当 `execution.protection.enabled=true` 时，`EXEC_PROTECTIVE_ORDER_MISSING` 必须为 `0`。
   - 含义：`SL` 挂单失败、附着超时、盈利保护更新 `SL` 失败，都属于主链硬失败。
   - 对应报告字段：`runtime_assess.metrics.protective_order_missing_count`

2. `S5 观测项`
   - `PROTECTION_REFRESH`
   - `EXEC_TP_ATTACH_FAILED`
   - `PROFIT_PROTECTION_UPDATE`
   - 当前用途：确认保护链路是否真的被触发、`TP` 是否存在偶发挂单失败、盈利保护是否发生过更新。
   - 当前仅做 metrics / warn，不作为总评阻断项。

3. `非主链项`
   - 保护单是否显著改善 `realized_net_per_fill`
   - `break-even` / `trailing` 的收益归因
   - per-order 级别保护，而不是 symbol 总持仓级保护
   - 这些属于后续策略质量评估，不进入当前 `S5` 强门禁。
